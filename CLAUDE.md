# meter — engineering guide

Read `docs/VISION.md` (product) and `docs/ARCHITECTURE.md` (system design + decision record) first.
This file defines how we write code here. It **scopes** the global personal standards: the Effect-TS
rules apply to TypeScript only; the backend is Rust and has its own rules below.

## Golden rules

- **No shortcuts. Built for the long term.** Correct schemas, real migrations, heavy tests.
- **The ledger is sacred.** Money/credits are never lost, double-counted, or overspent. Every credit
  movement is an immutable double-entry transaction; balances are derived. All ingest is idempotent.
- **No AI footprint.** Commits, PRs, comments, and docs read as ordinary human work. No AI-attribution
  trailers or "generated by" lines anywhere.

## Rust (backend) standards

- Edition 2021+, `#![forbid(unsafe_code)]` in every crate unless a crate documents why it must opt out.
- **Errors as typed channels.** Tagged error enums via `thiserror`; return `Result<T, DomainError>`.
  No `unwrap()`/`expect()`/`panic!` outside tests and `main` startup. No swallowing errors.
- **Money is never a float.** Use integer minor units / fixed-point (`rust_decimal` or an explicit
  `Money`/`Credit` newtype). No `f64` anywhere near currency or credits.
- **Linear control flow.** Prefer early returns and `match` over nested `if/else`. Exhaustive `match`
  on domain enums — no catch-all `_` that hides new variants.
- SQL via `sqlx` with compile-time-checked queries where practical; all schema changes are migrations.
  Follow the OpenAI Postgres discipline: no full-table-rewrite migrations on hot tables; backfills are
  batched and rate-limited; statement timeouts on the hot path.
- `clippy` clean (`-D warnings`) and `rustfmt` enforced in CI. Public items on library crates documented.
- Tests: unit + property tests (`proptest`) for ledger invariants; integration tests against real
  Postgres/ClickHouse (testcontainers). The no-overdraft and idempotency invariants are property-tested.

## TypeScript (SDK + dashboard) standards

Follow the global personal standards strictly:
- Effect-TS idioms (`Effect`, `Layer`, `Context.Tag`, `Schema`, `Match`, `pipe`); errors as typed
  channels; `Schema` for all boundary validation; no `as` casts at boundaries.
- **Function guards, not `===` string checks.** No `if/else` chains. **No ternaries** — use `Match` or
  named helpers. Predicates `isX`/`hasX`/`canX`.
- Named exports only; explicit return types on exported functions; `import type`; no `@ts-ignore`; no
  default exports.
- Use the project design system; don't hand-roll primitives that exist.
- API types for the SDK/dashboard are **generated from the backend OpenAPI** — do not hand-write the
  wire contract.

## Quality gates (CI)

- Rust: `cargo fmt --check`, `cargo clippy -D warnings`, `cargo test`.
- Effect-TS code: run **agent-doctor** (JGalbss/agent-doctor) — keep the score high; fix the slop it flags.
- React/Next.js: run **react-doctor** (millionco/react-doctor) including its PR action.
- TS: typecheck + lint + tests.

## Contracts

The backend is the source of truth. It emits an OpenAPI spec; the TS SDK and dashboard client types are
codegen'd from it. Keep the wire protocol versioned.
