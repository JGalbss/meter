# meter — engineering guide

Read `docs/VISION.md` (product), `docs/ARCHITECTURE.md` (system design + decision record), and
`docs/adr/` (amendments) first. This file defines how we write code here. It **scopes** the global
personal standards: the Effect-TS rules apply to TypeScript (control plane, SDK, dashboard); the engine
is Rust and has its own rules below.

## Golden rules

- **No shortcuts, ever — enterprise quality.** Never make a short-term fix that sacrifices long-term
  outcomes. Treat everything you touch as potential tech debt; never keep legacy/dead code or outdated
  patterns — fix or replace them, push the code forward. Correct schemas, real migrations, full e2e
  tests, no stubs left unfinished, no skipped tests, no "temporary" workarounds. Verify with real
  commands/tests, never by assertion.
- **UI uses the design system, always.** The dashboard is built on the shadcn preset `b1z2hUjZ5c`
  (Dropbox aesthetic) with transitions.dev animations — never hand-roll primitives or ad-hoc
  transitions. See the `meter-design-system` skill in `.claude/skills/`.
- **The ledger is sacred.** Money/credits are never lost, double-counted, or overspent. Every credit
  movement is an immutable double-entry transaction; balances are derived. All ingest is idempotent.
  **Money-truth lives only in the Rust engine** — the control plane never computes money (ADR 0001).
- **Events are immutable; edits are append-only.** Never mutate an event or ledger entry in place.
  "Edit an event" = `amend_event` (a new version + a delta posting); "kill a failed run" = `void_run`
  (reverse its holds/settles). The audit trail is perfect; the UX feels mutable (ADR 0002).
- **Be atomic. Organize well. Do not cram.** This codebase will be large — keep it navigable:
  - One concept per file; small, focused modules; clear module trees with `mod.rs`/`index.ts` that only
    wire and re-export. If a file is growing past one responsibility, split it.
  - A function/type/module has one clear purpose and a clear name. Extract shared logic on the 2nd–3rd
    repeat; no premature abstraction, no copy-paste of non-trivial logic.
  - Atomic commits: one coherent change each, green (`fmt`/`clippy`/tests) before commit.
- **No AI footprint.** Commits, PRs, comments, and docs read as ordinary human work. No AI-attribution
  trailers or "generated by" lines anywhere.

## Rust (engine) standards

- Edition 2021, `#![forbid(unsafe_code)]` in every crate unless a crate documents why it must opt out.
- **Errors as typed channels.** Tagged error enums via `thiserror`; return `Result<T, DomainError>`.
  No `unwrap()`/`expect()`/`panic!` outside tests and `main` startup. No swallowing errors.
- **Money is never a float.** Use the `Money`/`Credit` newtypes (`rust_decimal::Decimal`). No `f64`
  anywhere near currency or credits.
- **Linear control flow.** Prefer early returns and exhaustive `match` over nested `if/else`; no
  catch-all `_` that hides new enum variants.
- SQL via `sqlx`; all schema changes are migrations. Follow the OpenAI Postgres discipline: no
  full-table-rewrite migrations on hot tables; backfills batched and rate-limited; statement timeouts on
  the hot path.
- `clippy` clean (`-D warnings`) and `rustfmt` enforced in CI. Public items on library crates documented.
- Tests: unit + property tests (`proptest`) for ledger invariants; integration tests against real
  Postgres/ClickHouse (testcontainers). The no-overdraft and idempotency invariants are property-tested
  and run against every `LedgerBackend` as a shared conformance suite.

## TypeScript (control plane, SDK, dashboard) standards

Follow the global personal standards strictly:
- Effect-TS idioms (`Effect`, `Layer`, `Context.Tag`, `Schema`, `Match`, `pipe`); errors as typed
  channels; `Schema` for all boundary validation; no `as` casts at boundaries. The control plane uses
  **Effect + Drizzle** over its Postgres config schema.
- **Function guards, not `===` string checks.** No `if/else` chains. **No ternaries** — use `Match` or
  named helpers. Predicates `isX`/`hasX`/`canX`.
- Named exports only; explicit return types on exported functions; `import type`; no `@ts-ignore`; no
  default exports (Next.js pages/layouts excepted via Biome override).
- Use the project design system; don't hand-roll primitives that exist.
- The engine RPC client (`ts-proto`/connect) is generated from `proto/`; the dashboard's control-plane
  client types are generated from the control-plane OpenAPI. Do not hand-write wire contracts.

## Contracts

Two surfaces, two generators, no hand-mirrored types:
- **engine ⇄ control-plane:** protobuf (Buf) → `prost` (Rust) + `ts-proto`/connect (TS). Buf
  breaking-change check in CI.
- **control-plane ⇄ dashboard/customer:** OpenAPI emitted by the control plane → typed client codegen.

## Quality gates (CI)

- Rust: `cargo fmt --check`, `cargo clippy -D warnings`, `cargo test` (+ `nextest`/`proptest`).
- Effect-TS code: run **agent-doctor** (`JGalbss/agent-doctor`) — keep the score high; fix flagged slop.
- React/Next.js: run **react-doctor** (`millionco/react-doctor`) incl. its PR action; Lighthouse budgets.
- TS: typecheck + Biome lint + tests.
- `proto/`: Buf lint + breaking-change check.
