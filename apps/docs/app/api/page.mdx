# API reference

meter exposes two HTTP surfaces:

- **Engine** (`http://localhost:8080`) — the Rust data plane. Ledger, pricing, enforcement, events,
  invoices. This is where money-truth lives.
- **Control plane** (`http://localhost:8090`) — the TypeScript configuration API. Organizations,
  agents, API keys, alert rules, webhooks, notifications. Computes no money.

All bodies are JSON. UUIDs are strings. Credit amounts are decimal strings (never floats).

---

## Engine

Base path `/v1` (except `/health`). The engine derives all balances from the ledger; every mutating
call is idempotent on its key. Every error returns `{ "error": "<code>", "message": "<text>" }` — the
codes and their HTTP statuses are listed under [errors](/errors).

### Health

```
GET /health
```

Returns `{ "status": "ok" }` when the engine is serving.

### Accounts

```
POST /v1/accounts
```

Open an account. Body:

| Field | Type | Notes |
| --- | --- | --- |
| `org_id` | uuid | Owning organization. |
| `scope` | enum | Account scope (e.g. organization / session). |
| `no_overdraft` | bool | Defaults `false`. When `true`, the account can never go negative. |
| `parent_id` | uuid? | Optional parent (for child/session accounts). |

```
GET  /v1/accounts/{id}/balance     # settled + held balance
GET  /v1/accounts/{id}/entries     # the immutable entry history
```

```
POST /v1/accounts/{id}/grants
```

Add credits. Body: `{ "amount": "1000", "source": <CreditSource>, "idempotency_key"?: string }`.

### Reservations (reserve → settle / void)

```
POST /v1/reservations
```

Place a hold. Body:

| Field | Type | Notes |
| --- | --- | --- |
| `account` | uuid | Account to hold against. |
| `reservation_id` | uuid | Caller-supplied id (idempotent). |
| `amount` | decimal string | Credits to hold. |
| `limit` | enum | Limit class governing the hold. |

Returns a `ReserveOutcome` (allowed or denied with the limiting balance).

```
POST /v1/reservations/{id}/settle   # body: { "actual": "<credits>" } — resolve to actual cost
POST /v1/reservations/{id}/void     # release the hold without charge
```

### Leases (per-session budgets)

```
POST /v1/leases
```

Open a lease: a `Session` child account funded by a conserving transfer from the parent. Body:
`{ "parent": <uuid>, "amount": "<credits>" }`. Returns the child account. Refused with `409` if the
parent cannot cover the amount.

```
POST /v1/leases/{id}/close
```

Close a lease, returning its unused balance (`settled − held`) to the parent. Returns
`{ "returned": "<credits>" }`. Credits are conserved end to end.

### Usage (the core loop)

```
POST /v1/usage
```

Price token usage against the rate-card catalog, record an immutable event, and charge credits — in one
call. Body:

| Field | Type | Notes |
| --- | --- | --- |
| `org_id` | uuid | Owning organization. |
| `account` | uuid | Account to charge. |
| `model` | string | Model id resolved against the rate-card catalog. |
| `idempotency_key` | string | Required; replays are no-ops. |
| `run_id` | uuid? | Optional run grouping (enables `void_run`). |
| `usage` | object | Token counts: `input_uncached`, `cache_read`, `cache_write`, `output`, `reasoning`. |

Returns `{ credits, cogs_usd, customer_price_usd, event_id, charged, settled, available }`.

```
POST /v1/usage/{id}/amend
```

Correct a priced turn. The engine re-prices the event from the corrected usage dimensions and posts the
**difference** to the ledger — a charge if the new price is higher, a refund if lower — so the
correction stays money-true (idempotent on an optional key). This is distinct from
`POST /v1/events/{id}/amend`, which edits an event's properties only and never touches the ledger
([ADR 0009](https://github.com/JGalbss/meter/blob/main/docs/adr/0009-amend-delta-posting.md)).

### Catalog & simulation

The catalog is the dated, compiled-in set of provider rate cards (`CATALOG_AS_OF`). Read it to discover
model ids and their pricing, or simulate a model swap over a batch of usage before committing to it.

```
GET  /v1/catalog                # { as_of, models: [...] } — every catalogued model
GET  /v1/catalog/{model_id}     # the resolved RateCard for one model
POST /v1/simulate               # re-price a batch of events under a different model
```

`POST /v1/simulate` takes `{ current_model, proposed_model, events: [<usage dims>] }` and returns
`{ current_model, proposed_model, event_count, credits_current, credits_proposed, credit_delta }` — no
ledger writes, so it is safe to call from a pricing UI.

### Token-priced enforcement

When you want to reserve and settle in **token terms** rather than a credit amount, the engine prices
the estimate and the actuals against the catalog for you:

```
POST /v1/usage/reserve                       # hold sized to an estimated token usage for a model
POST /v1/usage/reservations/{id}/settle      # settle with the actual token usage
```

`reserve` body: `{ account, reservation_id, model, estimate: {<usage dims>}, limit, rate_card_id?,
run_id? }` → `{ outcome, reserved_credits }`. `settle` body: `{ model, actual: {<usage dims>},
rate_card_id? }` → `{ credits_charged, balance_after }`.

### Events (immutable, append-only edits)

```
POST /v1/events                 # record an event (idempotent on idempotency_key)
GET  /v1/events/{id}            # fetch latest non-voided version
POST /v1/events/{id}/amend      # body: { "properties": {...} } — record a corrected version
POST /v1/runs/{id}/void         # void_run: reverse a run's events and ledger effects
GET  /v1/accounts/{id}/events   # list events for an account
```

`POST /v1/events` body: `{ org_id, idempotency_key, event_time?, meter, account, run_id?, properties }`.

### Analytics, budgets, invoices, audit

```
GET /v1/accounts/{id}/usage-by-day   # daily credit usage from the ledger (Postgres)
GET /v1/accounts/{id}/budget         # budget status
GET /v1/accounts/{id}/invoice        # deterministic invoice summed from the ledger
GET /v1/audit                        # the admin audit log
```

Organization-level usage analytics are derived from the event store (ClickHouse). They reflect amends
and voids — an amended event counts once at its corrected version, and a voided run drops out:

```
GET /v1/orgs/{id}/usage-by-model     # per-model events, tokens, and credits (highest spend first)
GET /v1/orgs/{id}/usage-by-day       # daily event + credit totals
GET /v1/orgs/{id}/event-count        # { "count": <live recorded events> }
```

The full engine surface — every path, parameter, request body, response, and schema — is also
published as a **[generated reference](/api/engine)** rendered directly from the engine's committed
OpenAPI document (`crates/meter-api/openapi.json`, served at `GET /openapi.json` and the source of
truth for engine SDK codegen), so it always matches the live contract.

## Control plane

Base path `/v1`. Configuration only — the dashboard and customers talk to this surface for everything
that is not a money movement.

### Authentication & roles

When auth is enabled, every request except `/health` must carry `Authorization: Bearer <api-key>`.
API keys are role-scoped (RBAC), enforced by method and resource:

| Role | Can do |
| --- | --- |
| `viewer` | Read-only — any `GET`. |
| `member` | Reads plus ordinary writes (agents, alert rules, notifications). |
| `admin` | Everything, including credential management (`/v1/api-keys`, `/v1/webhooks`). |

Roles are ranked (`viewer < member < admin`), so a higher role satisfies any lower requirement. A key
minted without a role defaults to `admin`. Mint a scoped key with
`POST /v1/api-keys { "orgId", "name", "role": "member" }`. Insufficient role → `403`; missing/invalid
key → `401`.

### Organizations & agents

```
GET  /v1/organizations
POST /v1/organizations
GET  /v1/agents
POST /v1/agents
```

### API keys

```
GET  /v1/api-keys
POST /v1/api-keys
POST /v1/api-keys/{id}/revoke
```

### Alert rules

```
GET  /v1/alert-rules
POST /v1/alert-rules
POST /v1/alert-rules/evaluate         # run rule evaluation now
POST /v1/alert-rules/{id}/enabled     # toggle a rule
```

### Webhooks

```
GET  /v1/webhooks
POST /v1/webhooks
POST /v1/webhooks/{id}/enabled
GET  /v1/webhook-deliveries           # delivery log (signed payloads)
```

### Notifications

```
GET  /v1/notifications
POST /v1/notifications
POST /v1/notifications/{id}/read
POST /v1/notifications/{id}/ack
```

---

> **Contracts.** Both HTTP surfaces are generated, never hand-mirrored. The control plane emits an
> **OpenAPI 3.1** document at `GET /openapi.json` — every request *and* response body comes from the
> same Effect `Schema` that validates the request, types the repository, and is returned by the handler
> (named, `$ref`'d component schemas) — and the dashboard's control-plane client types are **generated**
> from it (`openapi-typescript`), with CI failing on drift. The engine emits its own **OpenAPI** document
> the same way (served at `GET /openapi.json`, committed at `crates/meter-api/openapi.json` behind a
> drift gate); both generated references on this site render from those committed specs.
>
> The engine also exposes a **gRPC** surface for the engine ⇄ control-plane path: the protobuf contract
> lives in `proto/` (Buf module `meter.v1`), the engine's Rust types and service stubs are generated
> from it (`prost`/`tonic`), and CI enforces `buf lint` plus a breaking-change check against `main`.
> The control plane currently consumes the engine over its REST surface; generating the TypeScript gRPC
> client from the same protobuf module (`ts-proto`/connect) is the remaining binding on that path.

The full control-plane surface — every path, parameter, request body, response, and schema — is also
published as a **[generated reference](/api/control-plane)** rendered directly from that OpenAPI
document, so it always matches the live contract.
