# Self-host

meter is self-hostable end to end. The production-style stack is four services: **Postgres**
(money-truth), **ClickHouse** (usage events), the **engine**, and the **control plane**.

## Docker Compose

```bash
docker compose -f deploy/docker-compose.yml up --build
```

This starts:

| Service | Port | Role |
| --- | --- | --- |
| `postgres` | 5432 | Ledger (money-truth). |
| `clickhouse` | 8123 | Usage events + audit (system of record). |
| `engine` | 8080 | HTTP data plane. Applies ledger + ClickHouse migrations on boot. |
| `control-plane` | 8090 | Config API. Applies its config migrations on boot. |
| `dashboard` | 3000 | Operator console (Next.js). Set `DASHBOARD_PASSWORD` / `DASHBOARD_SESSION_SECRET`. |

The engine applies Postgres ledger migrations **and** ClickHouse event migrations at startup; the
control plane applies its config migrations at startup. No manual migration step is required for a
fresh stack.

## Kubernetes (Helm)

For production, deploy with the Helm chart in `deploy/helm/meter`:

```bash
helm install meter deploy/helm/meter \
  --set engine.image.repository=ghcr.io/you/meter-engine --set engine.image.tag=0.0.0 \
  --set controlPlane.image.repository=ghcr.io/you/meter-control-plane --set controlPlane.image.tag=0.0.0
```

It deploys the engine, control plane, and dashboard, with Postgres and ClickHouse either in-cluster
(default) or external. The engine is stateless — scale it with `engine.replicas`; the dashboard is
toggleable (`dashboard.enabled`). Point at managed data stores with
`postgres.enabled=false` / `clickhouse.enabled=false` and `engine.databaseUrl` / `engine.clickhouseUrl`.

## Engine configuration

| Variable | Required | Default | Notes |
| --- | --- | --- | --- |
| `METER_DATABASE_URL` | yes | — | Postgres connection string (money-truth). |
| `METER_CLICKHOUSE_URL` | yes | — | ClickHouse HTTP endpoint (events). |
| `METER_LISTEN_ADDR` | no | `0.0.0.0:8080` | HTTP bind address. |
| `METER_CREDIT_VALUE` | no | `0.000001` | Cash value (USD) of one credit. |

## Control-plane configuration

| Variable | Required | Default | Notes |
| --- | --- | --- | --- |
| `METER_CONTROL_PLANE_DATABASE_URL` | yes | — | Postgres connection string (config schema). |
| `METER_CONTROL_PLANE_PORT` | no | `8090` | HTTP port. |
| `METER_ENGINE_URL` | yes | — | Engine base URL the control plane calls. |
| `METER_EVALUATION_INTERVAL_SECONDS` | no | `60` | Alert-rule evaluation cadence. |

## Why two stores

Money and events have different access patterns, so they live in different stores
([ADR 0003](/concepts)):

- **Postgres** holds the ledger and audit log. Strong transactional guarantees, row-level locking on
  the hot path, no overdraft.
- **ClickHouse** holds usage events. High-volume append, fast analytical rollups (`usage-by-day`,
  per-model), versioned reads for the append-only edit model.

Postgres is the single source of money-truth; ClickHouse is the single source of event-truth. Neither
duplicates the other.

## Health checks

```bash
curl http://localhost:8080/health        # engine — liveness (static)
curl http://localhost:8080/health/ready  # engine — readiness (pings Postgres + ClickHouse)
curl http://localhost:8090/health        # control plane — liveness (static)
curl http://localhost:8090/health/ready  # control plane — readiness (pings Postgres)
```

`/health` is static liveness (the process is up) — independent of the stores, so a transient database
blip never trips a restart loop. `/health/ready` returns `200` only when the service's stores answer
(otherwise `503`), so it's the probe a load balancer or Kubernetes should gate traffic on. The Helm
chart wires liveness to `/health` and readiness to `/health/ready`.

## Production notes

- Back up Postgres as your money-truth of record; ClickHouse can be rebuilt from raw event ingest.
- Run the engine behind your own auth/ingress; the control plane issues and verifies API keys.
- Migrations are forward-only and applied on boot — deploy the engine before pointing traffic at it.

## API keys & tenant isolation

The control plane authenticates every request (except `/health`) with a bearer API key and enforces
both a **role** and a **scope** (ADR 0007):

- **Role** — `viewer` (read-only), `member` (writes), or `admin` (manage keys + webhooks).
- **Scope** — `platform` keys may act across **every** organization (this is what the dashboard /
  operator console uses); `org` keys are confined to the **one** organization that owns them.

Isolation is enforced on every org-scoped route: an `org` key that targets another organization gets
`403` on reads and creates and `404` on by-id mutations (the row is invisible to it). Listing or
creating organizations, and minting a `platform` key, are **platform-only** — an `org`-scoped admin
cannot escalate. Enable auth with `METER_REQUIRE_AUTH=true` (off in local dev).

Mint a key in the dashboard (**API keys → New API key**, which offers the scope) or over the API:

```bash
# An org-scoped key (the secure default) for a tenant's own automation:
curl -s $CP/v1/api-keys -H "authorization: Bearer $PLATFORM_KEY" \
  -d '{"orgId":"<org-uuid>","name":"tenant ci","role":"member","scope":"org"}'
```

New keys default to `scope: "org"`; provision the dashboard's key as `platform`. **Upgrade note:** the
`scope` migration backfills any pre-existing keys to `platform`, so existing integrations keep working —
re-issue them as `org` keys to tighten access. Postgres row-level security is planned as
defense-in-depth on top of this app-level enforcement (ADR 0007).

## Private / air-gapped deployment

meter runs entirely inside a private VPC with no public exposure and no runtime egress:

- **Keep the APIs internal.** Leave `ingress.controlPlaneHost` / `ingress.engineHost` unset to keep the
  engine and control plane reachable only in-cluster, or set them with an **internal** ingress class
  (an internal load balancer) so SDKs reach them over private DNS. The dashboard can stay internal the
  same way.
- **Bring your own data stores.** Point at managed Postgres and ClickHouse in your account with
  `postgres.enabled=false` / `clickhouse.enabled=false` and `engine.databaseUrl` /
  `engine.clickhouseUrl` — nothing leaves the VPC.
- **Mirror the images.** The `meter-engine` / `meter-control-plane` / `meter-dashboard` images publish
  to GHCR; mirror them into your private registry and set each `*.image.repository`. No pull from the
  public internet is needed at runtime.
- **No outbound for pricing.** The hosted rate-card catalog is a **compiled-in dated snapshot**
  (`CATALOG_AS_OF`) — pricing resolves in-process with no network calls. (The optional scheduled
  scraper is the only component that would reach out, and it is off by default.)
- **Secrets.** Database credentials and the dashboard session secret come from a Kubernetes Secret (or
  your external secret manager via the chart's existing references), never baked into an image.

The result is a closed deployment: money-truth in your Postgres, events in your ClickHouse, images from
your registry, and no runtime dependency on anything outside the VPC.
