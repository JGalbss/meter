# SDKs & integrations

## Strategy

- **Base typed clients** (TypeScript, Python, Go, …) are **generated by [Stainless](https://www.stainless.com)**
  from the engine's OpenAPI spec — not hand-maintained. The engine API is the single source of truth.
- **Adapters** are thin, hand-written wrappers that auto-instrument the major AI clients — extract token
  usage, emit it to meter, and govern runs (reserve → settle, void on failure). They sit on top of the
  base client and are the durable, hand-owned part.

## Two layers (and which is generated)

```
provider call ──▶ ADAPTER (hand-written, per language) ──▶ BASE CLIENT (Stainless-generated) ──▶ engine
                  normalize usage, auto-patch                transport, auth, retries, types
```

- The **base client** is generated by Stainless from the engine OpenAPI — transport, auth, retries,
  pagination, and types for every endpoint. One per language, regenerated on every spec change, so it
  never drifts from the API. This is the cheap, broad layer.
- The **adapter** is the differentiated value and is **hand-written per language**: it captures usage
  from a provider's response (or auto-patches the provider client), normalizes it to meter's token
  dimensions, and feeds the base client. **Stainless does not generate adapters** — they are bespoke
  instrumentation, so "adapters in language X" is real per-language work, not a codegen flip.

## Base client (Stainless)

The engine emits its **OpenAPI** spec from the axum routes via `utoipa` (`crates/meter-api/openapi.json`,
drift-checked in CI) — the single source of truth. The Stainless config lives at
[`sdks/stainless/openapi.stainless.yml`](../sdks/stainless/openapi.stainless.yml) and targets
**TypeScript, Python, Go, Java, Kotlin, and Ruby**; resources map the engine's tags to ergonomic
surfaces (`client.accounts.open`, `client.reservations.reserve`, `client.usage.meter`, …).

### Regenerating

1. Create a Stainless org and load `sdks/stainless/openapi.stainless.yml` into Stainless Studio (it
   validates and refines the config; confirm package names against each registry).
2. Connect the Stainless GitHub app / `stl` CLI; add the `STAINLESS_TOKEN` CI secret.
3. CI regenerates and publishes the base clients on every change to `crates/meter-api/openapi.json`, so
   they never drift from the engine API.

> External dependency: generation runs on Stainless's platform (org + token required) — it cannot run
> from this repo alone. The config and the spec are the parts owned here.

## Adapter roadmap (per language)

Adapters ship today in **TypeScript and Python** (Anthropic, OpenAI, Gemini/Vertex, Bedrock, LangChain,
Vercel AI — plus `withRun` governance and first-class auto-patch wrappers). Other languages get the
Stainless base client immediately; their adapters are added on demand, prioritized by where the agent
ecosystem lives (Go next is the likely candidate).

## Adapters (`sdks/typescript/src/adapters`)

Provider-agnostic and **structurally typed** (no dependency on the provider SDKs, so they keep working
across provider versions). Available today:

| Adapter | Covers |
|---|---|
| `anthropicUsage` | Anthropic / Claude + the Claude Agent SDK |
| `openaiUsage` | OpenAI (chat / responses) |
| `vercelAiUsage` | Vercel AI SDK (both token namings) |
| `geminiUsage` | Google Gemini / Vertex (`usageMetadata`) |
| `bedrockUsage` | AWS Bedrock Converse |
| `langchainUsage` | LangChain / LangGraph (`usage_metadata`) |
| `recordModelUsage(client, …)` | emit normalized usage as a meter event |
| `meteredCall(client, …, extract, call)` | wrap a provider call and record its usage |
| `withRun(client, …)` | reserve → settle run governance; auto-voids a failed run |

```ts
import { MeterClient, meteredCall, openaiUsage } from "@meter/sdk";

const meter = new MeterClient({ baseUrl: process.env.METER_URL! });
const completion = await meteredCall(
  meter,
  { orgId, account, model: "gpt-x", idempotencyKey: requestId, runId },
  (r) => openaiUsage(r.usage),
  () => openai.chat.completions.create({ /* … */ }),
);
```

All adapters ship in both the TypeScript and Python SDKs.

### Auto-patch a provider client

First-class auto-patch wrappers monkey-patch a provider client so **every** call is metered with no
call-site changes — `patchAnthropic` / `patchOpenAI` (TS) and `patch_anthropic` / `patch_openai`
(Python). Each returns an `Unpatch` to restore the original method.

```ts
import { MeterClient, patchAnthropic } from "@meter/sdk";

const meter = new MeterClient({ baseUrl: process.env.METER_URL! });
const unpatch = patchAnthropic(meter, anthropic, { orgId, account }); // mode: "charge" | "record"
await anthropic.messages.create({ model: "claude-opus-4-8", messages, max_tokens: 1024 });
unpatch();
```

The model and a per-call idempotency key are derived automatically; `onError` makes metering fail-open
(the provider response still returns); the client is duck-typed, so no provider package is imported.
